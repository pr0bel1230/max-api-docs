Что защищает аудио- и видеозвонки MAX от прослушки, перехвата
и несанкционированного подключения.

## Кратко: что защищено

| Слой | Протокол | Защита | Расшифровка без ключа? |
|------|----------|--------|----------------------|
| Сигналинг (команды звонка) | WSS (WebSocket over TLS) | TLS 1.3 | Нет |
| Push-уведомления (opcode 137) | WSS | TLS 1.3 | Нет |
| HTTP API звонков | HTTPS (TLS) | TLS 1.3 | Нет |
| Аудио/видео поток | SRTP over DTLS | DTLS-SRTP | Нет |
| TURN relay | SRTP over DTLS | DTLS-SRTP | Нет |

**Медиа и сигналинг защищены на разных уровнях разными ключами.**
Даже расшифровав сигналинг (TLS MITM), вы не расшифруете аудио —
ключи DTLS-SRTP согласуются напрямую между участниками через
DTLS handshake, который не проходит через signaling сервер.

## Цепочка аутентификации

Чтобы подключиться к signalling звонка, нужно пройти три уровня
аутентификации. Как на КПП с тремя пропусками:

```
MAX access_token ──→ opcode 158 ──→ call token ($...)
                                       │
                                       ↓
                              HTTP: auth.anonymLogin ──→ session_key + uid
                                       │                      │
                                       ↓                      ↓
                              HTTP: startConversation     VCP (из push 137)
                                       │                      │
                                       ↓                      ↓
                              ┌─ signaling token ─────────────┘
                              │   + userId
                              │   + conversationId
                              ↓
                    videowebrtc.okcdn.ru/ws2
```

### 1. MAX access token

Токен сессии web.max.ru. Получается через OAuth/login на сайте.
Привязан к конкретному пользователю MAX (userId). Без него нельзя
получить call token (opcode 158).

### 2. Call token (opcode 158)

Выдаётся сервером MAX (`ws-api.oneme.ru`) по запросу с MAX access token.
Начинается с `$`. Живёт ограниченное время (token_lifetime_ts).
Нужен для `auth.anonymLogin` — входа в инфраструктуру звонков.

### 3. Session (HTTP API звонков)

`auth.anonymLogin` на `calls.okcdn.ru` создаёт сессию звонков:
- `uid` — внутренний signaling ID (`910111054239`)
- `session_key` — ключ для всех последующих запросов к HTTP API
- `external_user_id` — ваш MAX userId (`3260455`)

**Именно на этом уровне формируется привязка:** signaling uid → MAX user.

### 4. Signaling token

Выдаётся в двух сценариях, и оба дают разные токены:

| Сценарий | Кто выдаёт | Формат | Получатель |
|----------|------------|--------|------------|
| Входящий звонок | Push 137 → VCP.tkn | Закодирован в VCP | Тот, **кому** звонят |
| Исходящий звонок | `startConversation` | `response.token` | Тот, **кто** звонит |

Оба токена содержат:
- Кому принадлежит токен (userId в URL)
- К какому звонку (conversationId)
- Срок действия (et — expire timestamp)

**Токен звонящего не подходит для принимающего, и наоборот.**
Каждый токен авторизует только одну роль в одном звонке.

### 5. WebSocket handshake (videowebrtc.okcdn.ru)

При подключении к signaling сервер проверяет:

```
token (URL) + userId (URL) + conversationId (URL) + entityType
```

Если хоть один параметр не соответствует — сервер разрывает соединение
с ошибкой `"invalid-token"`. Подтверждено экспериментально:
неверный userId → `"invalid‑token"`, истёкший token → `"invalid‑token"`,
завершённый звонок → `"conversation-ended"`.

## Транспортная защита (WSS/TLS)

Всё, что касается звонков, идёт только через зашифрованные соединения:

| Сервер | Протокол | Сертификат |
|--------|----------|------------|
| `ws-api.oneme.ru` | WSS (TLS) | Let's Encrypt / Cloudflare |
| `calls.okcdn.ru` | HTTPS (TLS) | Let's Encrypt / Cloudflare |
| `videowebrtc.okcdn.ru` | WSS (TLS) | Let's Encrypt / Cloudflare |

**Ни один из серверов не использует кастомный certificate pinning.**
Это стандартная практика — pinning есть только у банковских приложений
и некоторых платёжных систем. MAX использует стандартную цепочку
доверия ОС.

> **Что это значит:** в корпоративной среде с MDM-сертификатом
> (MITM-прокси) TLS-трафик MAX может быть расшифрован. Это не
> уязвимость MAX — это свойство TLS как такового в окружении,
> где атакующий контролирует доверенные корневые сертификаты.

**Даже с TLS MITM защита медиа не снимается** — см. ниже.

## Защита медиа (DTLS-SRTP)

Аудио- и видеопотоки в WebRTC шифруются протоколом DTLS-SRTP.
Это отдельный уровень, независимый от TLS.

### Как это работает

```
Участник A                    Участник B
    │                             │
    ├── DTLS handshake ──────────→│
    │←── DTLS handshake ──────────┤
    │  (обмен сертификатами +     │
    │   согласование ключей)      │
    │                             │
    ├── SRTP (аудио, зашифровано)─→│
    │←── SRTP (аудио, зашифровано)─┤
```

Ключи DTLS-SRTP:
1. Согласуются **напрямую между участниками** через DTLS handshake
2. Передаются **по тому же каналу**, что и медиа (UDP)
3. **Не проходят через signalling сервер**
4. **Не видны в WSS-трафике** — даже если расшифрован signalling

### Почему DTLS-SRTP не расшифровать

Даже при полном контроле над сетевым трафиком (локальная сеть,
TLS MITM, контроль TURN сервера):

- Ключи создаются в DTLS handshake, который идёт UDP-P2P
- Атакующий должен участвовать в DTLS handshake, чтобы знать ключи
- Для участия в DTLS нужно, чтобы одна из сторон звонка отправила
  DTLS-пакет атакующему, а не настоящему участнику
- Это требует **подмены ICE кандидатов** в signalling
- Но signalling зашифрован WSS — подменить ICE кандидаты нельзя
- Замкнутый круг: чтобы подменить ICE → нужно расшифровать WSS →
  DTLS ключи всё равно не получить (они не в WSS)

## P2P топология

MAX использует топологию `DIRECT` — это Peer-to-Peer (P2P).

**Что это значит:**

```
    ┌─────────────────┐
    │ signalling сервер│  ← только координация
    └────────┬────────┘
             │
    ┌────────┴────────┐
    │  WSS (команды)  │  ← зашифровано TLS
    └────────┬────────┘
             │
    ┌────────┴────────┐
    │   Участник A    │←── UDP (аудио/видео, DTLS-SRTP) ──→│  Участник B    │
    └─────────────────┘                                      └─────────────────┘
```

- Медиа идёт напрямую между участниками (UDP)
- Сервер **не имеет доступа к медиапотоку**
- Сервер не может записать или перенаправить аудио/видео
- Если P2P невозможен (симметричный NAT) — используется TURN relay
- **TURN тоже шифрован DTLS-SRTP** — relay не видит содержимое

### TURN relay (форсированный)

Если P2P не проходит (оба участника за симметричным NAT),
медиа идёт через TURN-серверы `turn:155.212.x.x:19302`.

TURN — это просто ретранслятор. Он:
- Получает зашифрованные UDP-пакеты
- Пересылает их другому участнику
- **Не имеет ключей расшифровки**
- **Не может читать аудио/видео**
- **Не может записывать разговор**

## Подключение к своему звонку с другого устройства

VCP-токен из push 137 (или signaling token из `startConversation`)
выдаётся **участнику звонка**, а не **устройству**.

### Что это значит на практике

- Вы можете открыть звонок в браузере на MacBook
- Подключиться к тому же звонку с Python скрипта на том же MacBook
- Подключиться к тому же звонку с другого компьютера
- Подключиться к тому же звонку с телефона

**Сервер пускает все подключения**, если они аутентифицированы
одним и тем же signaling токеном. Максимум участников — 1500
(`participantsLimit: 1500`).

### Что видит звонящий

Другой участник звонка **не видит**, что вы подключились второй раз:

- В signalling нет сообщения «кто ещё смотрит»
- ServerHello при новом подключении не рассылается остальным
- `registered-peer` приходит только когда участник публикует
  свой WebRTC peer — второй WS не обязан это делать
- Активный участник видит только своего `peerId` и собеседника

**Но:** если второй WS начнёт отправлять SDP/ICE (полноценный
WebRTC accept), то:
- Сервер может разослать `registered-peer` другим участникам
- Это может быть заметно (появление нового ICE кандидата)

**Пассивное наблюдение** (просто слушать ServerHello и SDP)
— сервер не сигнализирует о нём другим участникам.

### Что можете делать вы

С подключением второго WS к своему звонку можно:
- Наблюдать ServerHello с параметрами звонка
- Получать SDP offer/answer (видеть WebRTC параметры)
- Получать ICE candidates (видеть сетевые пути участников)
- **Нельзя слушать аудио** — DTLS-SRTP ключи есть только у браузера

Для прослушки аудио через второй WS нужно:
1. Получить SDP offer (это мы умеем)
2. Ответить SDP answer с WebRTC peer (нужен aiortc или браузер)
3. Пройти DTLS handshake
4. Расшифровывать SRTP пакеты

Это технически возможно, но выходит за рамки простого наблюдения.

## Подключение к чужому звонку

Теперь главный вопрос: можно ли подключиться к звонку, где
вы не являетесь участником?

### Как получить доступ

Для подключения к signalling нужно:
```
wss://videowebrtc.okcdn.ru/ws2
  ?userId=<signaling_uid жертвы>
  &conversationId=<UUID звонка>
  &token=<signaling_token жертвы>
```

**userId + conversationId + token должны быть валидны и составлять
одну связку.** Три независимых параметра, которые проверяются вместе.

### Откуда взять эти данные?

| Параметр | Откуда берётся | Доступность |
|----------|---------------|-------------|
| `userId` | VCP из push 137 или `auth.anonymLogin` | Только получатель звонка |
| `conversationId` | Push 137 или `startConversation` | Только участники |
| `token` | VCP.tkn или `startConversation` | Только участники |

**Все три параметра передаются только через:**

1. **Push 137** — WSS (TLS), приходит только получателю звонка
2. **startConversation** — HTTPS (TLS), виден только звонящему
3. **ServerHello** — WSS (TLS), виден только подключившимся к signalling

**Ни один из этих каналов не является публичным.**
Эфир не вещается, broadcast не используется.

### Push 137 — не broadcast

Важный момент: push **opcode 137** сервер отправляет **не всем**,
а адресно — только тому MAX userId, **на кого звонят**. Ваш
access token привязан к вашему userId (3260455). Вы получаете
только свои пуши. Чужие 137 вы не увидите никогда.

Даже если вы слушаете ту же WS (`ws-api.oneme.ru`), сервер
не отправляет вам чужие уведомления — он знает, какой userId
за каким access token закреплён.

### Что будет, если подключиться со случайным UUID

- `userId` не совпадает с тем, кому выдан токен → `invalid-token`
- `conversationId` не существует → `invalid-token`
- `conversationId` существует, звонок завершён → `conversation-ended`
- `token` чужой или истёкший → `invalid-token`
- `userId` правильный, но звонок не ваш → `invalid-token`

**Все проверки на стороне сервера, клиентской валидации нет.**

## Что видно в локальной сети

Если вы находитесь на одной сети с участником звонка (Wi-Fi,
Ethernet) и перехватываете трафик, вот что вы увидите:

### Видно

| Что | Пример | Информативность |
|-----|--------|-----------------|
| DNS запросы | `ws-api.oneme.ru`, `videowebrtc.okcdn.ru` | «Кто-то использует MAX» |
| HTTPS соединения | Куда и когда | «В момент звонка» |
| Размер и периодичность пакетов | Паттерны трафика | «Аудио- или видеозвонок» (приблизительно) |
| STUN binding requests | `stun:155.212.x.x:19302` | «Идёт WebRTC соединение» |
| TURN relay трафик | UDP пачки на `155.212.x.x:19302` | «Размер медиапотока» |
| ICE кандидаты (публичные IP) | В STUN сообщениях | Публичные IP участников |

### НЕ видно

| Что | Почему |
|-----|--------|
| conversationId звонка | Внутри зашифрованного WSS |
| Кто звонит/кому звонят | Внутри зашифрованного WSS |
| Signaling token | Внутри зашифрованного WSS |
| VCP данные | Внутри зашифрованного WSS |
| SDP offer/answer | Внутри зашифрованного WSS |
| Аудио/видео содержимое | DTLS-SRTP — ключи на устройстве |
| DTLS рукопожатие | Идёт P2P, ключи не выходят за пределы handshake |

### Пассивный анализ трафика

Единственное, что можно извлечь из перехваченного трафика без
TLS ключей — это паттерны:

- **Всплеск UDP до STUN/TURN** — начало звонка (ICE gathering)
- **Регулярные UDP пакеты ~50-100 байт** — аудиозвонок (Opus ~40-80 kbps)
- **Регулярные UDP пакеты ~1000-1200 байт** — видеозвонок (VP8/VP9/H264)
- **Прерывание UDP потока + WSS сообщение** — завершение звонка

**Паттерны не дают содержимого разговора**, только факт звонка и
примерный тип (аудио/видео).

## Гипотетические сценарии атак

### 1. TLS MITM (корпоративный прокси)

**Условия:** атакующий контролирует корневой CA на устройстве жертвы
(MDM, корпоративный ноутбук).

**Что даёт:**
- Расшифровка WSS трафика (signaling)
- Просмотр ServerHello, SDP, ICE candidates
- Получение conversationId, userId, token

**Что НЕ даёт:**
- Аудио/видео (DTLS-SRTP ключи не в signalling)
- Ключи DTLS-SRTP (они согласуются P2P в UDP, не проходят через сервер)

**Можно ли подставить SDP?** Технически — да, если расшифрован WSS.
Можно изменить SDP offer до того, как он дойдёт до получателя.
Но это требует активного вмешательства (MITM на уровне WebSocket),
а не пассивного наблюдения.

**Реалистичность:** низкая для массовой прослушки, возможна для
целевой атаки в контролируемой среде. Не даёт доступа к содержимому
разговора.

### 2. ARP-spoofing в локальной сети

**Условия:** атакующий в той же локальной сети (Wi-Fi).

**Что даёт:**
- Факт звонка (DNS, STUN, размеры пакетов)
- Публичные IP участников (из STUN)

**Что НЕ даёт:**
- TLS ключи для расшифровки WSS
- DTLS ключи для расшифровки аудио
- conversationId или token
- Содержимое разговора

**Реалистичность:** ARP-spoofing в 2026 году работает только
в legacy-сетях без защиты уровня L2 (802.1X, Dynamic ARP Inspection).
Даже при успешном спуфинге даёт только метаданные.

### 3. Атака на TURN сервер

**Условия:** атакующий контролирует TURN сервер или стоит на пути
к нему (BGP hijack).

**Что видит TURN:**
- Зашифрованные UDP пакеты (DTLS-SRTP)
- IP адреса отправителя и получателя
- Размеры пакетов

**Что НЕ видит:**
- Ключи DTLS-SRTP (согласуются P2P, не через TURN)
- Содержимое аудио/видео (зашифровано)
- Кто участники звонка (ID только в WSS)

**Реалистичность:** BGP hijack — сложная атака. Даже с контролем
TURN сервера содержимое недоступно.

### 4. Участие в DTLS handshake (активная атака)

**Условия:** атакующий хочет вклиниться в DTLS handshake между
участниками.

**Проблема:** DTLS использует самоподписанные сертификаты WebRTC.
В SDP offer/answer передаются fingerprint'ы этих сертификатов
(через WSS). Если fingerprint в SDP не совпадает с фактическим
сертификатом в DTLS — handshake падает с ошибкой.

```
SDP offer от А: fingerprint=SHA-256 AAAA...

Атакующий: «я изменю fingerprint на свой»
Но: SDP зашифрован WSS — изменить нельзя
```

**Замкнутый круг:** чтобы подменить fingerprint — нужно расшифровать
WSS. Но даже с расшифрованным WSS, атакующий должен:
1. Перехватить SDP offer реального участника
2. Подменить fingerprint на свой
3. Отправить модифицированный SDP второму участнику
4. При этом оба участника должны не заметить подмены

**Реалистичность:** практически нулевая для пассивной атаки.
Требует активного MITM на двух каналах одновременно (WSS + UDP).

### 5. Session hijack (кража токена)

**Условия:** атакующий получил VCP-токен жертвы (например, через
XSS, фишинг, локальный вредонос).

**Что даёт:**
- Подключение к signalling звонка жертвы
- Просмотр ServerHello, SDP, ICE

**Что НЕ даёт:**
- Прослушка аудио/видео (нужен DTLS-SRTP)
- Доступ к другим звонкам (токен одноразовый, на конкретный звонок)
- Доступ к MAX аккаунту в целом

**Важно:**
- VCP-токен живёт столько же, сколько звонок (et в VCP — expire timestamp)
- Токен привязан к одному conversationId
- После завершения звонка токен бесполезен

**Реалистичность:** кража VCP — непрактичный вектор. Для XSS нужно
найти уязвимость в web.max.ru. Для фишинга — убедить пользователя
передать VCP (который скрыт в protobuf и не отображается в UI).
Вредонос на устройстве — проще записать аудио через микрофон ОС.

### 6. WebRTC IP leak

**Условия:** атакующий хочет узнать реальный IP участника звонка.

**Что происходит:** при установке WebRTC соединения браузер
отправляет ICE candidates, которые включают:
- `host` — локальный IP (192.168.x.x)
- `srflx` — публичный IP (через STUN)
- `relay` — TURN relay (IP TURN сервера)

Эти IP можно увидеть:
- В signalling (зашифрован WSS — нельзя)
- В STUN binding responses (открытый UDP — можно)
- SDP offer — там есть fingerprint и IP кандидатов (в WSS — нельзя)

**Публичный IP участника** можно получить через STUN, даже не
подключаясь к звонку. STUN binding request к `stun:turn-XX.okcdn.ru:19302`
вернёт ваш публичный IP. Это не специфика MAX — это свойство STUN
протокола.

**Локальный IP** (192.168.x.x) не виден за пределами локальной сети.

### Сводка

| Сценарий | Сложность | Даёт содержимое? | Реалистичность |
|----------|-----------|-----------------|----------------|
| Пассивный перехват (ARP) | Низкая | Нет | Высокая |
| TLS MITM (корп. CA) | Средняя | Нет (только сигналинг) | Средняя (для целевых атак) |
| Кража VCP | Высокая | Нет | Низкая |
| BGP hijack TURN | Очень высокая | Нет | Крайне низкая |
| Активный DTLS MITM | Экстремальная | Возможно теоретически | Near-zero |
| Физический доступ к устройству | Средняя | Да | Высокая |

**Единственный надёжный способ прослушать звонок — получить
физический доступ к устройству одного из участников.**

## Выводы

1. **Звонки MAX защищены на всех уровнях:**
   - WSS/TLS — транспортная защита
   - DTLS-SRTP — защита медиа независимыми ключами
   - P2P — сервер не имеет доступа к аудио/видео
   - Адресная рассылка push — чужие звонки не видны

2. **Нельзя прослушать звонок через локальную сеть:**
   - Вся сигнализация зашифрована
   - Медиа зашифровано отдельными ключами
   - Ключи DTLS-SRTP не проходят через сервер

3. **Нельзя подключиться к чужому звонку без его токена:**
   - VCP/token выдаётся конкретному участнику
   - Сервер проверяет token + userId + conversationId
   - Push 137 приходит только получателю

4. **Можно подключиться к своему звонку с другого устройства:**
   - Токен выдаётся участнику, а не устройству
   - Сервер не ограничивает количество WS соединений
   - Другие участники не видят «лишних» подключений
   - Но аудио так не прослушать — нужен DTLS

5. **Наибольшая угроза — компрометация устройства, а не сети:**
   - Вредонос на телефоне/ноутбуке может записать аудио через ОС
   - Фишинг access token MAX даёт доступ к сообщениям, но не к звонкам
   - Корпоративный MITM-прокси раскрывает метаданные, но не аудио
