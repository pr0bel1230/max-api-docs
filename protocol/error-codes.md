# Коды ошибок

MAX API использует два уровня сигнализации ошибок:
поле `cmd` в заголовке фрейма и поле `error`/`message` в payload.

## Уровень 1: cmd в заголовке

| cmd | Значение |
|-----|----------|
| 1 | Успешный ответ (ACK) |
| 3 | Ошибка |

**Важно:** `cmd=1` не гарантирует, что операция выполнена успешно.
Некоторые опкоды (например, 65 — MSG_TYPING) всегда возвращают `cmd=1`,
игнорируя содержимое payload. Единственный надёжный способ проверки —
запросить состояние после операции.

## Уровень 2: поле error в payload

При `cmd=3` payload содержит информацию об ошибке:

```json
{
  "error": "proto.payload",
  "message": "Field requirement failed: messageIds",
  "title": "Ошибка валидации",
  "localizedMessage": "Ошибка валидации"
}
```

### Стандартные поля ошибки

| Поле | Тип | Описание |
|------|-----|----------|
| `error` | string | Код ошибки (машинное имя) |
| `message` | string | Описание ошибки (на английском) |
| `title` | string | Заголовок (на русском) |
| `localizedMessage` | string | Локализованное описание (на русском) |

## Известные ошибки

### proto.payload — ошибка валидации payload

Возникает, когда в запросе не хватает обязательных полей.

| Сообщение | Опкоды | Причина |
|-----------|--------|---------|
| `"Field requirement failed: messageIds"` | 74 (GET_STATS) | Не указан `messageIds` |
| `"Field requirement failed: time"` | 177 (GET_USERINFO) | Не указан `time` |
| `"Field requirement failed: chatId"` | 86 (CHAT_SHOW) | Не указан `chatId` |

### proto.opcode — неизвестный или недопустимый опкод

Возникает при отправке опкода, который не поддерживается как запрос.

**Важно:** Вызов такого опкода может **закрыть WebSocket-соединение**.
Это происходит с некоторыми push-only опкодами (например, 132 — NOTIF_PRESENCE).

### Другие ошибки

| Сообщение | Опкоды | Причина |
|-----------|--------|---------|
| `"Key: Queue is not configured"` | 92 (CHAT_ACTIVITY) | `startTime = 0` |
| `"Key: chat.not-found"` | 86 (CHAT_SHOW) | `show: false` без дополнительных полей |
| `"userIds can't be empty"` | 77 (CHAT_OPERATION) | Пустой `userIds` |
| `"invalid token"` | 19 (LOGIN) | Неверный или истёкший токен |

### Нестандартные ошибки

Некоторые опкоды возвращают ошибки в собственном формате вместо
стандартного:

| Опкод | Ответ при ошибке |
|-------|-----------------|
| 68 (MSG_FWD) | `cmd=3` с пустым payload |

## Ошибки соединения

### WebSocket close codes

При закрытии WebSocket-соединения сервер может отправить код:

| Код | Описание |
|-----|----------|
| 1000 | Нормальное закрытие |
| 1006 | Аномальное закрытие (обрыв соединения) |
| 1008 | Нарушение политики (возможно, невалидный запрос) |

### Таймаут сессии

- Сервер закрывает неактивное соединение через ~30-60 секунд
- PING (opcode 1) с `{"interactive": true}` продлевает сессию
- После обрыва нужно создать новое соединение и пройти INIT → LOGIN

### Сброс токена

- После ~30-50 LOGIN-ов с одним токеном за короткий промежуток
  сервер может сбросить токен
- Симптом: новое соединение закрывается сразу после INIT
- Решение: получить свежий токен из web.max.ru
