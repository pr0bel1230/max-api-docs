# Сообщения: отправка, получение, редактирование, удаление

Управление сообщениями — основной функционал MAX API.
Все операции над сообщениями используют отдельные опкоды.

## Структура сообщения

Сообщение в MAX имеет следующую структуру (в ответах сервера):

```json
{
  "id": 116762160362694583,
  "time": 1781704393993,
  "type": "USER",
  "sender": 3260455,
  "cid": 35,
  "text": "текст сообщения",
  "elements": [],
  "attaches": [],
  "reactionInfo": {}
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int/string | Уникальный ID сообщения (число или строка) |
| `time` | int | Unix timestamp в миллисекундах |
| `type` | string | Тип: `USER`, `CHANNEL` |
| `sender` | int/null | ID отправителя (user_id). `null` для каналов |
| `cid` | int | Client ID — идентификатор для дедупликации |
| `text` | string | Текст сообщения |
| `elements` | array | Rich-элементы (см. [elements.md](elements.md)) |
| `attaches` | array | Вложения: `PHOTO`, `VIDEO`, `FILE`, `UNSUPPORTED`, `SHARE`, `CONTROL`, `CALL`; для каналов может содержать `keyboard` (INLINE_KEYBOARD) (см. [files.md](files.md)) |
| `reactionInfo` | object | Реакции на сообщение |
| `stats` | object | Статистика (только CHANNEL): `{"views": 1009}` |
| `status` | string | Статус: `EDITED` |
| `updateTime` | int | Timestamp редактирования (мс, при `status: "EDITED"`) |
| `options` | int | Опции. Для CHANNEL: `1` (назначение не установлено) |
| `deleted` | bool | Присутствует только у сервисных уведомлений |
| `link` | object | Репост/пересылка (только CHANNEL). Содержит `type: "FORWARD"` и данные исходного сообщения (см. раздел «Репосты (FORWARD)») |

**Примечание:** `id` может передаваться как число или как строка
в разных контекстах. При сравнении лучше приводить к строке.

### Типы сообщений

| Тип | Описание | sender |
|-----|----------|--------|
| `USER` | Обычное сообщение от пользователя | int (user_id) |
| `CHANNEL` | Пост в канале | `null` (автор не раскрывается) |

**Примечание:** Типы `SERVICE` и `SYSTEM` не существуют — в исследованном
трафике не встречено ни одного сообщения с такими типами. Системные
события (например, `botStarted`) передаются через вложения типа `CONTROL`
внутри обычных `USER`-сообщений. Подробнее — в [files.md](files.md).

### Каналы (CHANNEL)

Сообщения из каналов имеют особенности:
- `sender` всегда `null`
- Поле `stats` содержит количество просмотров: `{"views": 1009}`
- Могут содержать вложение с `keyboard` (INLINE_KEYBOARD) для комментариев
- Поддерживают редактирование (`status: "EDITED"`, `updateTime`)
- `options: 1` — назначение не установлено
- Могут содержать поле `link` с репостом другого сообщения (см. «Репосты (FORWARD)»)

### Репосты (FORWARD)

При репосте (пересылке) сообщения из одного канала в другой сообщение
содержит поле `link` с типом `"FORWARD"`:

```json
{
  "id": "116765257400649050",
  "type": "CHANNEL",
  "text": "",
  "link": {
    "type": "FORWARD",
    "chatId": -69110159553957,
    "chatAccessType": "PUBLIC",
    "chatName": "ПРОПЕЛЛЕР",
    "chatLink": "https://max.ru/id7712038455_biz",
    "chatIconUrl": "https://i.oneme.ru/i?r=...",
    "message": {
      "id": "116764968146969003",
      "time": 1781692018844,
      "text": "Текст исходного сообщения...",
      "type": "CHANNEL",
      "attaches": []
    }
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `link.type` | string | Всегда `"FORWARD"` |
| `link.chatId` | int | ID канала-источника |
| `link.chatAccessType` | string | Тип доступа: `"PUBLIC"` (найдено), другие возможны |
| `link.chatName` | string | Название канала-источника |
| `link.chatLink` | string | Ссылка на канал (https://max.ru/...) |
| `link.chatIconUrl` | string | URL иконки канала-источника |
| `link.message` | object | Исходное сообщение (id, time, text, type, attaches) |

**Особенности:**
- Поле `link` найдено только в сообщениях типа `CHANNEL`
- Репост поддерживается только между каналами (не между DIALOG/CHAT)
- Текст репоста может быть пустым (`text: ""`) — содержимое исходного сообщения
  отображается через `link.message`
- FORWARD — единственный найденный тип `link`. Другие типы (например, `REPLY`)
  не обнаружены в исследованном трафике

### INLINE_KEYBOARD (кнопка в каналах)

Сообщения в каналах могут содержать кнопку для комментариев.
Кнопка передаётся как вложение с полем `keyboard`:

```json
{
  "attaches": [
    { "_type": "PHOTO", ... },
    {
      "keyboard": {
        "buttons": [[
          {
            "text": "💬 Прокомментировать →",
            "type": "OPEN_APP",
            "webApp": "https://max.ru/id302500195328_2_bot",
            "contactId": 193584517,
            "payload": "comment_channel_-68335643047333"
          }
        ]]
      }
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `keyboard.buttons` | array[array] | Массив рядов кнопок. Каждый ряд — массив кнопок |
| `buttons[][].text` | string | Текст на кнопке |
| `buttons[][].type` | string | Тип: `"OPEN_APP"` (найдено) |
| `buttons[][].webApp` | string | URL мини-приложения для открытия |
| `buttons[][].contactId` | int | ID контакта-бота для комментариев |
| `buttons[][].payload` | string | Payload кнопки (идентификатор канала/чата) |

**Особенности:**
- Кнопка не является вложением с `_type` — она располагается во вложении
  без `_type`, но с полем `keyboard`
- В одном сообщении может быть несколько вложений: медиа + keyboard
- Тип `OPEN_APP` открывает мини-приложение (бота комментариев)
- Другие типы кнопок (`CALLBACK`, `URL`, `LOCATION`) не обнаружены

### Rich-элементы (elements[])

Поле `elements` — массив аннотаций к тексту сообщения.
Каждый элемент описывает форматированный или интерактивный
фрагмент (ссылку, жирное начертание, курсив, подчёркивание)
через offset+length в тексте.

**Внимание:** `elements[]` **не возвращается** в ответе на MSG_SEND
и MSG_EDIT. Он присутствует только в ответе GET_HISTORY и в
push-уведомлениях о новых сообщениях.

Подробная документация — в [elements.md](elements.md).

**Известные типы:**

| type | Описание | attributes |
|------|----------|------------|
| `LINK` | Гиперссылка | `url` — полный URL |
| `STRONG` | Жирное начертание | — |
| `EMPHASIZED` | Курсив | — |
| `UNDERLINE` | Подчёркивание | — |

**Назначение:** elements[] позволяет клиентам рендерить текст
с интерактивными элементами без парсинга текста регулярными
выражениями.

## MSG_SEND (opcode 64)

Отправка нового сообщения в чат.

### Запрос

```json
{
  "chatId": 7268926,
  "message": {
    "text": "Привет!",
    "cid": 1734567890123,
    "elements": [],
    "attaches": []
  },
  "notify": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `message.text` | string | Текст сообщения |
| `message.cid` | int | Client ID — `int(time.time() * 1000)` или произвольный |
| `message.elements` | array | Rich-элементы (пустой массив = обычный текст) |
| `message.attaches` | array | Вложения (см. [files.md](files.md)) |
| `notify` | bool | Отправить уведомление участникам чата |

**Client ID (cid):** Используется для дедупликации сообщений на клиенте.
Обычно — текущий timestamp в миллисекундах. Два сообщения с одинаковым
`cid` от одного отправителя считаются одним.

### Ответ

```json
{
  "chatId": 7268926,
  "message": {
    "id": 116762203424780659,
    "time": 1781704393993,
    "text": "Привет!",
    "sender": 3260455,
    "type": "USER",
    "cid": 1734567890123,
    "attaches": [],
    "reactionInfo": {}
  },
  "unread": 1,
  "mark": 1781704393993
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата, куда отправлено сообщение |
| `message` | object | Полный объект сообщения |
| `unread` | int | Количество непрочитанных сообщений в чате |
| `mark` | int | Watermark — timestamp последнего прочитанного сообщения (мс) |

**Особенности:**
- Поле `elements` **отсутствует** в ответе — сервер не возвращает
  элементы форматирования при отправке. Чтобы получить elements[],
  запросите историю (GET_HISTORY).
- `mark` — это водяной знак чата, используемый для отсчёта
  непрочитанных сообщений. Совпадает с `mark` из GET_CHATS.
- `message.id` может быть строкой или числом в зависимости от
  транспорта.

### Отправка стикеров

Стикеры можно отправлять двумя способами:

**Способ 1:** Поля `stickerId` и `setId` в теле сообщения (cmd=1 — принято):

```json
{
  "chatId": 7268926,
  "message": {
    "text": "",
    "cid": 1734567890123,
    "elements": [],
    "attaches": [],
    "stickerId": 138,
    "setId": 100517
  },
  "notify": true
}
```

**Способ 2:** Вложение STICKER в `attaches[]` — требуется серверный ID стикера:

```json
{
  "chatId": 7268926,
  "message": {
    "text": "",
    "cid": 1734567890123,
    "elements": [],
    "attaches": [{
      "_type": "STICKER",
      "stickerType": "STATIC",
      "setId": 100517,
      "stickerId": 138
    }]
  },
  "notify": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `stickerId` | int | ID стикера в наборе |
| `setId` | int | ID набора стикеров |
| `stickerType` | string | Тип стикера: `"STATIC"` |

При получении стикер отображается в `attaches[]` (см. [Структура сообщения](#структура-сообщения)).

### Отправка геолокации

Геолокация передаётся через поля `latitude` и `longitude` в теле сообщения:

```json
{
  "chatId": 7268926,
  "message": {
    "text": "Я тут",
    "cid": 1734567890123,
    "latitude": 55.7558,
    "longitude": 37.6173
  },
  "notify": true
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `latitude` | double | Широта |
| `longitude` | double | Долгота |

**Особенности:**
- Вложение `_type: "LOCATION"` в `attaches[]` требует поле `latitude`
  (не `lat`). Альтернативный вариант — указать координаты напрямую
  в сообщении.
- Способ отображения геолокации в полученном сообщении не исследован.

## MSG_EDIT (opcode 67)

Редактирование отправленного сообщения.

### Запрос

```json
{
  "chatId": 7268926,
  "messageId": "116766060130147953",
  "text": "Новый текст сообщения",
  "elements": [],
  "attachments": []
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `messageId` | string | ID сообщения (строка) |
| `text` | string | Новый текст |
| `elements` | array | Rich-элементы (обязательно, пустой массив для обычного текста) |
| `attachments` | array | Вложения (обязательно) |

### Ответ

```json
{
  "message": {
    "sender": 3260455,
    "reactionInfo": {},
    "updateTime": 1781708687628,
    "id": "116766060130147953",
    "time": 1781708681185,
    "text": "Новый текст сообщения",
    "type": "USER",
    "status": "EDITED",
    "cid": -1781708680996,
    "attaches": []
  }
}
```

- Ответ — полный объект сообщения
- Появляется поле `status: "EDITED"`
- Появляется поле `updateTime` — timestamp редактирования (отличается от `time`)
- Поля `elements` и `attachments` в запросе **обязательны** — без них сервер возвращает ошибку

## MSG_DELETE (opcode 66)

Удаление сообщения.

### Запрос

```json
{
  "chatId": 7268926,
  "messageIds": [116762203424780659],
  "forMe": false
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `messageIds` | array[int/string] | Массив ID сообщений для удаления |
| `forMe` | bool | `false` — удалить у всех, `true` — только у себя |

### Ответ

```json
{
  "chatId": 7268926,
  "messageIds": [116762203424780659]
}
```

### Особенности

- **⚠️ ОПАСНО — Каскадное удаление (WATERMARK BUG):** MSG_DELETE (66) с `forMe: false`
  может удалять **все сообщения до определённой точки**, а не только указанные в `messageIds`.
  Механизм:

    1. **CHAT_ACTIVITY (opcode 92)** при успешном выполнении устанавливает на сервере
       внутреннюю метку `lastReadTimestamp` («водяной знак», watermark).
    2. **MSG_DELETE (66) c `forMe: false`**, вызванный сразу после 92, удаляет
       **ВСЕ сообщения, созданные до этой метки**, а не только указанные в `messageIds`.
    3. Сообщения, отправленные **после** вызова 92, не затрагиваются (кроме целевого).

    **Триггер каскада**: вызов 92 + 66. Если не вызывать 92 — DELETE работает корректно
    (удаляет ровно указанные сообщения). Этот баг воспроизводится стабильно:
    3 «прогревочных» итерации, затем 92 → DELETE = каскад каждый раз.

    **Единственное исключение**: если 92 вызывается с диапазоном времени **в будущем**
    (`startTime/endTime > now`), каскад не происходит — водяной знак устанавливается
    в будущем и не перекрывает ни одно сообщение.

    **Рекомендации:**
    - **Никогда не использовать `forMe: false` на реальных чатах.**
      Даже если кажется, что запрос безопасен.
    - Для тестов используйте `forMe: true` — удаляет только у отправителя.
    - Не вызывайте opcode 92 (CHAT_ACTIVITY) перед `forMe: false`.
    - Если нужно безопасно протестировать DELETE — создайте тестовый чат.
- **Множественное удаление:** `messageIds` может содержать несколько ID —
  они будут удалены одним запросом
- **`forMe`:** `true` = мягкое удаление (как "удалить у себя" в Telegram).
  Сообщение исчезает только у отправителя, у других участников остаётся
- **`forMe: false`:** удаление у всех участников чата. Необратимо.
  Сообщения полностью исчезают с сервера — восстановление через API невозможно.
- **Повторное удаление:** при попытке удалить уже удалённое сообщение
  сервер возвращает `cmd=3`
- **Проверка:** единственный надёжный способ убедиться, что удаление
  сработало — запросить историю (opcode 49) и убедиться, что сообщения
  нет. Сервер отвечает `cmd=1` даже на запросы, которые не выполняют
  ожидаемого действия.

### Push-уведомления

При удалении сообщения другим участником чата приходит:
- **opcode 142** (NOTIF_MSG_DELETE) — уведомление об удалении
- **opcode 140** (NOTIF_MSG_DELETE_RANGE) — уведомление о массовом удалении

Подробнее в [push.md](push.md).

## MSG_REACT_SET (opcode 178)

Поставить реакцию на сообщение.

### Запрос

```json
{
  "chatId": 7268926,
  "messageId": 116765779164748382,
  "reaction": {
    "reactionType": "EMOJI",
    "id": "👍"
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `messageId` | int | ID сообщения |
| `reaction.reactionType` | string | Тип реакции: `EMOJI`. Только объекты, не строки. |
| `reaction.id` | string | Эмодзи (Unicode-символ, например `"👍"`, `"😍"`). Плоская строка не работает. |

### Ответ

```json
{
  "reactionInfo": {
    "counters": [
      {"count": 1, "reaction": "👍"}
    ],
    "yourReaction": "👍",
    "totalCount": 1
  }
}
```

Возвращает полный `reactionInfo` с обновлёнными счётчиками.
Для проверки также можно использовать GET_MESSAGE (71).

### Особенности

- Поле `reaction` **обязательно должно быть объектом** с `reactionType` и `id`.
  Передача строки (`"reaction": "👍"`) возвращает ошибку `cmd=3`:
  `"Expected map at 11"`.
- Ответ содержит `yourReaction` — эмодзи, которое только что поставили.

### Push-уведомление

При установке реакции сервер отправляет push **opcode 156** (NOTIF_REACTION):

```json
{
  "chatId": 7268926,
  "messageId": "116765779164748382",
  "reactionInfo": {
    "counters": [
      {"count": 1, "reaction": "👍"},
      {"count": 1, "reaction": "😍"}
    ],
    "yourReaction": "👍",
    "totalCount": 2
  }
}
```

Поле `yourReaction` присутствует только если реакция поставлена текущим пользователем.

## MSG_REACT_REMOVE (opcode 179)

Снять реакцию с сообщения.

### Запрос

```json
{
  "chatId": 7268926,
  "messageId": 116765779164748382
}
```

### Ответ

`cmd=1` (ACK).

## GET_REACTIONS (opcode 181)

Получение списка реакций на сообщение с указанием, кто поставил.

### Запрос

```json
{
  "chatId": 7268926,
  "messageId": "116765779164748382",
  "count": 100
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата (может быть отрицательным при unsigned overflow) |
| `messageId` | string | ID сообщения (строка) |
| `count` | int | Количество реакций в ответе |

### Ответ

```json
{
  "reactionInfo": {
    "counters": [
      {"count": 1, "reaction": "❤️"}
    ],
    "totalCount": 1
  },
  "reactions": [
    {
      "userId": 112572428,
      "reaction": "❤️"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `reactions[].userId` | int | ID пользователя, поставившего реакцию |
| `reactions[].reaction` | string | Эмодзи реакции |

### Отличие от GET_MESSAGE (71)

GET_MESSAGE возвращает только `reactionInfo` — агрегированные счётчики
по типам реакций. GET_REACTIONS (181) дополнительно возвращает массив
`reactions` с userId, показывая кто именно поставил каждую реакцию.

## GET_REACTIONS_BULK (opcode 180)

Получение реакций для **нескольких сообщений** одновременно.
В отличие от GET_REACTIONS (181) — для одного сообщения.

### Запрос

```json
{
  "chatId": 7268926,
  "messageIds": [
    "116358901737221942",
    "116358901747248950"
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `messageIds` | array[string] | Массив ID сообщений |

### Ответ

```json
{
  "messagesReactions": {
    "116358901737221942": {},
    "116359350248428856": {
      "counters": [{"count": 1, "reaction": "👍"}],
      "yourReaction": "👍",
      "totalCount": 1
    }
  }
}
```

- Ключи объекта `messagesReactions` — строковые ID сообщений
- Если на сообщении нет реакций — пустой объект `{}`
- `yourReaction` присутствует если реакция стоит от текущего пользователя

### Отличие от GET_REACTIONS (181)

GET_REACTIONS (181) возвращает детальный список `reactions: [{userId, reaction}]`
с указанием кто поставил. GET_REACTIONS_BULK (180) возвращает только агрегированные
счётчики, но зато для многих сообщений сразу.

## MSG_TYPING (opcode 65)

Отправка индикатора набора текста. Не выполняет никаких действий
над сообщениями.

### Запрос

```json
{
  "chatId": 7268926
}
```

### Ответ

`cmd=1` — всегда успех. Это не означает, что выполнена какая-либо

## CHAT_ACTIVITY (opcode 92)

Установка временной метки последнего посещения чата (watermark).
Используется для отсчёта непрочитанных — сервер помечает все сообщения
до указанного времени как прочитанные для текущего пользователя.

### Запрос

```json
{
  "chatId": 309052361,
  "startTime": 1781688781664,
  "endTime": 1781697221724
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `startTime` | int | Timestamp начала периода активности (мс) |
| `endTime` | int | Timestamp конца периода активности (мс) |

### Ответ

```json
{
  "chat": {
    "id": 309052361,
    "mark": 1781704393991,
    "members": {
      "6236697": 1781688781664,
      "3260455": 1781697221724
    },
    "modified": 1781686441819,
    "lastEventTime": 1781686441819,
    "status": "REMOVED",
    "type": "DIALOG"
  }
}
```

### Особенности

- Устанавливает водяной знак (`mark`) — все сообщения до этой метки
  считаются прочитанными
- `members` (внимание: это поле называется `members`, а не `participants`,
  в отличие от GET_CHATS / GET_CHAT_INFO) содержит `{userId: lastReadTimestamp}`
  для каждого участника чата
- `startTime`/`endTime` — временной диапазон активности пользователя.
  `endTime` определяет новую позицию watermark
- **⚠️ Безопасность:** Вызов CHAT_ACTIVITY перед MSG_DELETE с
  `forMe: false` может вызвать каскадное удаление (см. раздел
  [MSG_DELETE](#msg_delete-opcode-66))

Получение статистики сообщений (просмотры для каналов).

```json
  "messageIds": [116777569816950010]

```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата (канала) |
| `messageIds` | array[int] | Массив ID сообщений |

### Ответ

```json
{
  "stats": {
    "116777569816950010": {
      "views": 1028
```

### Особенности

## GET_MEDIA (opcode 51)

Получение сообщений с вложениями определённого типа (медиа-галерея).
Используется при открытии вкладки «Медиа» в чате.

### Запрос

```json
{
  "chatId": -72842705805946,
  "messageId": "116766023624784450",
  "attachTypes": ["PHOTO", "VIDEO"],
  "forward": 25,
  "backward": 25
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `messageId` | string | ID опорного сообщения, вокруг которого искать медиа |
| `attachTypes` | array[string] | Типы вложений: `PHOTO`, `VIDEO`, `FILE`, `UNSUPPORTED`, `AUDIO`, `SHARE` |
| `forward` | int | Количество сообщений с медиа вперёд от опорного |
| `backward` | int | Количество сообщений с медиа назад от опорного |

### Ответ

```json
{
  "messages": [
    { "sender": 36581684, "id": "116589720409760495", "time": ..., "text": "...", "type": "USER",
      "attaches": [{"_type": "PHOTO", "width": 810, "height": 420, "previewData": "...", ...}] }
  ]
}
```

Возвращает полные объекты сообщений, содержащие только те, у которых
есть вложения указанного типа. Массив `attaches` может содержать
несколько вложений.

### Особенности

- `attachTypes` содержит **один тип за раз** — клиент делает отдельные
  запросы для PHOTO+VIDEO, FILE, AUDIO, SHARE
- `messageId` — строка (ID сообщения как строка)
- Пагинация: `forward` / `backward` — количество сообщений **с медиа**,
  не общее. Если между двумя медиа-сообщениями 10 текстовых, они
  пропускаются.
- Опорное сообщение задаёт точку отсчёта — медиа до и после неё

### GET_MEDIA с type/query (GIPHY, GIF, STICKER)

GET_MEDIA (51) также используется для поиска GIF-изображений и стикеров
через внешние интеграции. Формат запроса отличается от стандартного:

```json
{
  "type": "GIPHY",
  "query": "cat",
  "limit": 3,
  "chatId": 7268926
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `type` | string | Тип поиска: `"GIPHY"`, `"GIF"`, `"STICKER"`, `"ANIMOJI"` |
| `query` | string | Поисковый запрос (для GIPHY/GIF) |
| `limit` | int | Количество результатов (макс. не установлен) |
| `chatId` | int | **Обязательно** — ID чата (без него — ошибка валидации) |

### Ответ (type/query)

```json
{
  "messages": [],
  "total": 62
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `messages` | array | Найденные медиа-объекты (может быть пустым) |
| `total` | int | Общее количество результатов поиска |

### Особенности

- Поле `chatId` **обязательно** — без него сервер возвращает `cmd=3`
  с ошибкой `"Field requirement failed: chatId"`
- Для GIPHY/GIF возвращает `total: 62` независимо от запроса —
  возможно, это ограничение сервера или заглушка
- Для STICKER/ANIMOJI также требуется `chatId`, поле `query` не нужно
- Полный формат объектов в `messages` не установлен — при тестировании
  возвращается пустой массив
- `cursor` для пагинации передаётся как строка (пустая строка для
  первого запроса), но на момент исследования не влияет на результат

## GET_HISTORY (opcode 49)

Получение истории сообщений чата.

### Запрос

```json
{
  "chatId": 7268926,
  "forward": 0,
  "backward": 10,
  "backwardTime": 0,
  "forwardTime": 0,
  "getChat": false,
  "from": 1734567890123,
  "getMessages": true,
  "interactive": false
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `backward` | int | Количество сообщений назад (до) |
| `forward` | int | Количество сообщений вперёд (после) |
| `backwardTime` | int | Сдвиг по времени назад (0 = от текущего) |
| `forwardTime` | int | Сдвиг по времени вперёд |
| `getChat` | bool | Включить информацию о чате в ответ |
| `from` | int | Timestamp, от которого начинать отсчёт |
| `getMessages` | bool | Вернуть массив сообщений |
| `interactive` | bool | Флаг для серверной оптимизации |

### Ответ

```json
{
  "messages": [
    {
      "id": 116762160362694583,
      "time": 1781704393993,
      "type": "USER",
      "sender": 3260455,
      "text": "текст",
      "attaches": [],
      "reactionInfo": {}
    }
  ],
  "chat": { ... }   // если getChat: true
}
```

### Пагинация

Для подгрузки более старых сообщений используйте `from` = timestamp
самого старого сообщения из предыдущего ответа.

**Важно:** `backward` работает **только в комбинации с `from`**.
Без поля `from` запрос с `backward > 0` возвращает пустой массив.
`forward`, напротив, работает без `from` (по умолчанию от `from=0`).

Примеры правильных запросов:

- Последние 30 сообщений: `{from: now_timestamp, backward: 30, forward: 0}`
- Первые 50 сообщений: `{from: 0, backward: 0, forward: 50}`
- Вокруг конкретного сообщения: `{from: msg_timestamp, backward: 10, forward: 10}`

### Удалённые сообщения

Удалённые сообщения **полностью отсутствуют** в ответе GET_HISTORY —
они не помечаются как `deleted`, а просто исчезают.
Это отличает MAX от Telegram, где удалённые сообщения приходят
с флагом deleted=true.

## SEARCH_MESSAGES (opcode 73)

Поиск сообщений в чате по текстовому запросу.

### Запрос

```json
{
  "query": "привет",
  "count": 5,
  "chatId": 7268926
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `query` | string | Текст для поиска |
| `count` | int | Количество результатов |
| `chatId` | int | ID чата для поиска (обязательно) |

### Ответ

```json
{
  "result": [
    {
      "message": {
        "sender": 3260455,
        "id": "116762245115485336",
        "time": 1781650468681,
        "text": "Саша, привет! ...",
        "type": "USER"
      }
    }
  ]
}
```

### Особенности

- **chatId обязателен** — без него сервер возвращает ошибку
- Поиск полнотекстовый — ищет по содержимому сообщений
- Возвращает только найденные сообщения (не чаты/контакты)
- Сообщения в ответе имеют ту же структуру, что и в GET_HISTORY

## MSG_SEARCH_GLOBAL (opcode 60)

Глобальный поиск сообщений по всем чатам. В отличие от SEARCH_MESSAGES (73),
не требует chatId и возвращает информацию о чате вместе с сообщениями.

### Запрос

```json
{
  "query": "test",
  "count": 3
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `query` | string | Поисковый запрос (обязательно) |
| `count` | int | Количество результатов (опционально) |

### Ответ

```json
{
  "result": [
    {
      "chat": {
        "owner": 0,
        "participantsCount": 2515,
        "access": "PUBLIC",
        "lastMessage": { ... }
      }
    }
  ],
  "total": 481
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `result` | array[object] | Массив результатов с информацией о чате |
| `total` | int | Общее количество найденных результатов |

### Особенности

- `query` обязателен — без него сервер возвращает `"Field requirement failed: query"`
  (на самом деле cmd=1 с пустым результатом, но логически ошибка)
- Возвращает информацию о чате с последним сообщением, а не только сообщения
- Поиск полнотекстовый по всем доступным чатам пользователя
- `total` показывает общее количество найденных результатов
- `count` опционален; при отсутствии сервер возвращает стандартное количество
- Сообщения в `result[i].message` имеют ту же структуру, что и в GET_HISTORY

## SEARCH_LIVE_STREAMS (opcode 62)

Поиск прямых эфиров (live streams) по идентификаторам чатов.
Возвращает данные о стримах для указанных чатов.

### Запрос

```json
{
  "chatIds": [-72842705805946],
  "text": ""
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatIds` | array[int] | Массив ID чатов для поиска стримов |
| `text` | string | Текст поиска (опционально) |

### Ответ

```json
{
  "liveStreams": [
    {
      "chatId": -72842705805946,
      "live": false,
      "canPublish": true,
      "pinned": false,
      "senderId": 0,
      "id": 0,
      "ownerId": 0,
      "status": "FINISHED"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `liveStreams` | array[object] | Массив данных о стримах для запрошенных чатов |
| `chatId` | int | ID чата |
| `live` | bool | Идёт ли прямой эфир сейчас |
| `canPublish` | bool | Может ли пользователь публиковать стрим |

### Особенности

- Используется для отображения иконки прямого эфира в списке чатов
- `live` + `status: "FINISHED"` означает, что стрим был, но завершён
- Для чатов без стримов возвращается пустой массив
- Не является поиском сообщений — это отдельный функционал прямых эфиров

## MSG_FWD (opcode 68) — пересылка сообщений (deprecated?)

Пересылка одного или нескольких сообщений. Возвращает `cmd=3`
(ошибка) — возможно, опкод больше не поддерживается или требует
специфического формата запроса, не обнаруженного в исследованном трафике.

**Запрос (предположительно):**
```json
{
  "chatId": 7268926,
  "messageIds": ["116762203424780659"]
}
```

**Ответ:** `cmd=3` (ошибка).

**Особенности:**
- Указан в таблице опкодов с `cmd=3`, что может означать неполный
  payload или неиспользуемый endpoint
- В Go-клиенте icyfalc0n/maxcalls отсутствует — опкод не реализован
- Для пересылки сообщений используйте FORWARD_MESSAGE (70)
