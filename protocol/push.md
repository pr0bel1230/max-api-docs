# Push-уведомления (cmd=0)

Сервер может отправлять сообщения без запроса — push-уведомления.
Они приходят асинхронно, в любое время, вперемешку с ответами
на запросы. Отличаются полем `cmd=0`.

**Важно:** Push-опкоды работают **только в одном направлении** —
сервер → клиент. Попытка отправить их как запрос (клиент → сервер)
приводит к ошибке `"Unknown opcode"` или разрыву соединения.

Исключения: опкоды **144** и **145** работают и как запросы
(см. [contacts.md](contacts.md)).

Все известные push-опкоды:

| Опкод | Название | Описание |
|-------|----------|----------|
| 128 | NOTIF_MSG_EDIT | Уведомление о редактировании сообщения |
| 129 | NOTIF_TYPING | Уведомление о наборе текста |
| 130 | UNKNOWN_130 | Подтверждение прочтения от другого участника |
| 1 | PING | Проверка соединения (heartbeat) |
| 132 | NOTIF_PRESENCE | Изменение статуса присутствия пользователя |
| 136 | NOTIF_ATTACH | Подтверждение загрузки файла |
| 137 | NOTIF_INCOMING_CALL | Уведомление о входящем звонке |
| 140 | NOTIF_MSG_DELETE_RANGE | Уведомление о массовом удалении |
| 142 | NOTIF_MSG_DELETE | Уведомление об удалении сообщения |
| 156 | NOTIF_REACTION | Уведомление о реакции на сообщение |


## 128 — NOTIF_MSG_EDIT

Уведомление о редактировании сообщения другим участником чата.
Приходит после подписки на чат (CHAT_SUBSCRIBE).

```
cmd=0 opcode=128
```

## 129 — NOTIF_TYPING

Уведомление о том, что собеседник печатает (или начал звонок).
Приходит после подписки на чат (CHAT_SUBSCRIBE). Перед входящим
звонком (opcode 137) приходит опережающее уведомление 129.

### Payload

```json
{
  "chatId": 7268926,
  "userId": 307889134
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |
| `userId` | int | ID пользователя, который набирает/звонит |

## 1 — PING (heartbeat)

Сервер периодически проверяет активность соединения. Клиент должен
отвечать, иначе соединение может быть разорвано.

**Push от сервера:**
```
cmd=0 opcode=1 payload={}
```

**Ответ клиента:**
```
cmd=0 opcode=1 payload={"interactive": true}
```

Опкод 1 не имеет `seq` (или seq не совпадает) — это не запрос-ответ,
а ping/push+ack от обоих участников.

## 132 — NOTIF_PRESENCE

Приходит при изменении статуса пользователя (онлайн/офлайн/печатает).

**Важно:** Это push-only опкод. Отправка как запрос (cmd=1) вызывает
ошибку `"proto.opcode"` и закрывает WebSocket-соединение.

### Payload

```json
{
  "presence": {
    "seen": 1781705573399,
    "status": 1
  },
  "userId": 307889134
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `presence.seen` | int | Timestamp последнего действия (мс) |
| `presence.status` | int | Статус: `1` — онлайн |
| `userId` | int | ID пользователя |

Возникает примерно через 7 секунд после того, как пользователь
совершил какое-либо действие (отправил сообщение, открыл чат и т.д.).
В контексте тестирования DELETE этот push был ошибочно принят
за подтверждение удаления.

## 136 — NOTIF_ATTACH

Подтверждение, что загруженный на сервер файл готов к прикреплению.

```
cmd=0 opcode=136
```

Используется в процессе отправки файлов:
1. FILE_UPLOAD (87) → получение URL
2. HTTP POST → загрузка файла
3. NOTIF_ATTACH (136) → подтверждение
4. MSG_SEND (64) → отправка сообщения с аттачем

Подробнее в [files.md](files.md).

## 130 — UNKNOWN_130 (NOTIF_READ / подтверждение прочтения)

Push-уведомление, подтверждающее что сообщения в чате прочитаны.
Приходит когда **другой** участник прочитал сообщения (не сам).

### Payload

```json
{
  "ver": 11,
  "cmd": 0,
  "opcode": 130,
  "payload": {
    "setAsUnread": false,
    "chatId": 309052361,
    "userId": 307889134,
    "mark": 1781890447978
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `setAsUnread` | bool | `false` — отметка о прочтении (не "не прочитано") |
| `chatId` | int | ID чата |
| `userId` | int | ID пользователя, который прочитал |
| `mark` | int | Timestamp отметки (мс) |

### Особенности

- Отличается от CHAT_READ (50): 130 — это **push**, а CHAT_READ — **запрос** на отметку
- Приходит на все соединения, подписанные на чат (CHAT_SUBSCRIBE)

## 142 — NOTIF_MSG_DELETE

Уведомление об удалении сообщения другим участником чата.

Приходит, когда кто-то другой удалил сообщение, и сервер уведомляет
об этом остальных участников.

## 140 — NOTIF_MSG_DELETE_RANGE

Уведомление о массовом удалении сообщений (диапазоном).

## 156 — NOTIF_REACTION

Уведомление о том, что кто-то поставил или снял реакцию на сообщение.
Подробнее в [messaging.md#msg_react_set-opcode-178](messaging.md#msg_react_set-opcode-178).

```json
{
  "chatId": 7268926,
  "messageId": "116765779164748382",
  "reactionInfo": {
    "counters": [{"count": 1, "reaction": "👍"}],
    "yourReaction": "👍",
    "totalCount": 1
  }
}
```

Поле `yourReaction` присутствует, если реакция стоит от текущего
пользователя (т.е. того, чей токен используется в соединении).
Приходит на каждое изменение реакций — как установку, так и снятие.

## Обработка push-уведомлений

При чтении ответов из сокета нужно фильтровать push-уведомления:

```python
while True:
    frame = recv_frame(sock)
    cmd = frame['cmd']
    opcode = frame['opcode']
    
    if cmd == 0:
        # Это push — обработать или пропустить
        handle_push(opcode, frame['payload'])
        continue
    
    if cmd == 1:
        # Это ответ на запрос
        process_response(opcode, frame['payload'])
        break
```

Поскольку push-уведомления не имеют `seq` (или он не совпадает
с запросом), сопоставление ответов с запросами через `seq`
автоматически их отфильтровывает.
