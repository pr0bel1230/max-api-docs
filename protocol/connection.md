# Управление сессией

Жизненный цикл соединения с MAX: установка → авторизация → обмен
запросами → закрытие.

Формат фрейма и способ подключения описаны в
[websocket.md](websocket.md) (JSON) и [tcp-protocol.md](tcp-protocol.md)
(MessagePack). Здесь — как управлять сессией: какой подход к соединению
выбрать, как сопоставлять ответы с запросами, что делать с
push-уведомлениями и таймаутами.

## Два подхода к соединению

### Подход 1: новое соединение на каждый запрос (НЕ РЕКОМЕНДУЕТСЯ)

Ранние скрипты делали так: открыть WebSocket → INIT → LOGIN → один
запрос → закрыть.

```python
def probe(opcode, payload):
    ws = connect()
    ws.send({"opcode": 6, "payload": {"deviceId": did}})
    ws.recv()                         # INIT OK
    ws.send({"opcode": 19, "payload": {"token": tok}})
    ws.recv()                         # LOGIN OK
    ws.send({"opcode": opcode, "payload": payload})
    result = ws.recv()
    ws.close()
    return result
```

**Проблемы:**
- **Сжигает токен.** 10-20 запросов = 10-20 LOGIN-ов. Сервер
  сбрасывает токен после ~30-50 LOGIN-ов в короткий промежуток.
- **Медленный.** Каждый раз TLS handshake (~200-500 мс) + полный цикл
  INIT+LOGIN.
- **Артефакты.** При закрытии соединения сервер может выполнять
  очистку сессии, что искажает поведение некоторых опкодов.

### Подход 2: постоянное соединение (РЕКОМЕНДУЕТСЯ)

Одно соединение на всю работу. Один INIT, один LOGIN, много запросов.

```python
ws = connect()
ws.send({"opcode": 6, "payload": {"deviceId": did}})
ws.recv()                             # INIT OK
ws.send({"opcode": 19, "payload": {"token": tok}})
ws.recv()                             # LOGIN OK

# 20 запросов на одном соединении
for _ in range(20):
    ws.send({"opcode": 64, "payload": {...}})
    ws.recv()

ws.close()
```

**Преимущества:**
- Один LOGIN на сессию — токен не сгорает
- В 5-10 раз быстрее — нет оверхеда на установку соединения
- Нет артефактов закрытия сессии

## seq-матчинг

При постоянном соединении ответы могут перемежаться с
push-уведомлениями (cmd=0). seq-матчинг — как понять, какой ответ
к какому запросу относится.

Клиент сам назначает `seq` каждому запросу. Сервер отвечает с тем же
`seq`:

```
→ seq=100 (запрос)
→ seq=101 (запрос)
← cmd=0 push   (игнорируем)
← cmd=1 seq=100 (ответ)
→ seq=102 (запрос)
← cmd=1 seq=101 (ответ)
← cmd=1 seq=102 (ответ)
```

**Реализация (универсально):**

> Функции `send_frame()` и `recv_frame()` — абстракции. Для WebSocket
> это `ws.send(json.dumps({...}))` / `json.loads(ws.recv())`,
> для TCP — `pack_frame()` / `recv_frame()` из
> [tcp-protocol.md](tcp-protocol.md).

```python
pending = {}      # {seq: результат}
next_seq = 0

def send(opcode, payload):
    global next_seq
    next_seq += 1
    pending[next_seq] = None
    send_frame({"seq": next_seq, "opcode": opcode, "payload": payload})

# Цикл получения — фильтруем push, матчим seq
while True:
    msg = recv_frame()
    if msg["cmd"] == 0:
        continue              # push-уведомление
    if msg["cmd"] == 1 and msg["seq"] in pending:
        pending.pop(msg["seq"])
        # обработать ответ
    elif msg["cmd"] == 3:
        pending.pop(msg["seq"], None)
        # обработать ошибку
```

**Различия транспортов:**

| Транспорт | seq | Замечание |
|-----------|-----|-----------|
| WebSocket | любое число 0-65535 | seq в заголовке JSON |
| TCP | seq % 256 (0-255) | seq в 2 байтах заголовка |

## Push-уведомления (cmd=0)

Сервер может отправлять сообщения без запроса клиента. Они приходят
с `cmd=0`, поле `seq` отсутствует.

| Опкод | Событие |
|-------|---------|
| 1 | PING (heartbeat, cmd=0 от сервера) |
| 128 | Уведомление о редактировании сообщения (NOTIF_MSG_EDIT) |
| 129 | Уведомление о наборе текста (NOTIF_TYPING) |
| 130 | Подтверждение прочтения (UNKNOWN_130/NOTIF_READ) |
| 132 | Изменение статуса присутствия (NOTIF_PRESENCE) |
| 136 | Подтверждение загрузки файла (NOTIF_ATTACH) |
| 137 | Уведомление о входящем звонке (NOTIF_INCOMING_CALL) |
| 140 | Массовое удаление сообщений (NOTIF_MSG_DELETE_RANGE) |
| 142 | Удаление сообщения (NOTIF_MSG_DELETE) |
| 156 | Реакция на сообщение (NOTIF_REACTION) |

Детали — в [push.md](push.md).

При постоянном соединении push приходят в тот же канал, что и ответы.
seq-матчинг отфильтровывает их из потока запрос-ответ.

## Ошибки

Сервер отвечает `cmd=3` при ошибке. Пример:

```json
{
  "cmd": 3,
  "opcode": 64,
  "payload": {
    "error": "proto.payload",
    "message": "invalid token"
  }
}
```

Стандартного набора кодов нет — поле `error` может различаться для
разных опкодов. Всегда проверяй наличие поля `error` в ответе (подробнее
в [error-codes.md](error-codes.md)).

## Таймауты

- Сервер закрывает неактивное соединение. Точное время не определено,
  но ~30-60 секунд простоя — и соединение сбрасывается.
- PING (opcode 1) с `{"interactive": true}` продлевает сессию.
- После обрыва нужно создать новое соединение и пройти INIT → LOGIN
  заново.

**Рекомендация:** для редких запросов проще каждый раз создавать
новое соединение. Для серии запросов — держать постоянное,
переподключаться при обрыве.
