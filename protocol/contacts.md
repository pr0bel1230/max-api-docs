# Контакты и профиль пользователя

Информация о пользователях и контактах.

## Профиль пользователя

Данные текущего пользователя возвращаются в ответе LOGIN (opcode 19)
в поле `profile`:

```json
{
  "profile": {
    "contact": {
      "id": 3260455,
      "names": [
        {"name": "Имя Фамилия", "type": "FULL_NAME"}
      ],
      "about": "статус пользователя",
      "phones": [
        {"number": "+71234567890", "type": "MOBILE"}
      ],
      "picture": {
        "url": "https://...",
        "base": "..."
      },
      "registrationDate": "..."
    },
    "settings": {
      "notifications": true,
      "theme": "light",
      ...
    },
    "subscription": {
      "expires": "...",
      "active": true,
      ...
    }
  }
}
```

### Contact

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | int | Уникальный ID пользователя |
| `names` | array | Массив имён с типами |
| `names[].name` | string | Имя |
| `names[].type` | string | `FULL_NAME`, `FIRST_NAME`, `LAST_NAME` |
| `about` | string | Статус ("недоступен" и т.д.) |
| `phones` | array | Номера телефонов |
| `picture` | object | Аватар: url и base64 миниатюра |

## GET_CONTACTS (opcode 32)

Получение информации о контактах по их ID.

### Запрос

```json
{
  "contactIds": [3260455, 3260456, 3260457]
}
```

### Ответ

```json
{
  "contacts": [
    {
      "id": 3260455,
      "names": [{"name": "Имя", "type": "FULL_NAME"}],
      "about": "статус",
      "phones": [{"number": "+71234567890", "type": "MOBILE"}],
      "picture": {"url": "https://...", "base": "..."}
    }
  ]
}
```

### Особенности

- Можно запрашивать несколько контактов за раз
- Если контакт не найден — вероятно, просто отсутствует в ответе
- Формат каждого контакта совпадает с форматом `contact` в профиле

## CONTACTS_PRESENCE (opcode 35)

Проверка статуса присутствия контактов (онлайн/офлайн).

### Запрос

```json
{
  "contactIds": [307889134, 6236697]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `contactIds` | array[int] | ID контактов для проверки |

### Ответ

```json
{
  "presence": {
    "307889134": {
      "seen": 1781705573399
    },
    "6236697": {
      "seen": 1781705461985
    }
  }
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `presence` | object | Ключи — ID пользователей (строки) |
| `presence[id].seen` | int | Timestamp последнего визита (мс) |

### Особенности

- Возвращает объект, где ключи — строковые ID пользователей
- Поле `seen` — время последней активности (unix timestamp в миллисекундах)
- Отсутствие пользователя в ответе ≠ он офлайн, скорее недоступна информация
- Отличается от push-уведомления NOTIF_PRESENCE (132), которое приходит асинхронно

## GET_BLOCKED (opcode 36)

Получение списка заблокированных пользователей.

### Запрос

```json
{
  "status": "BLOCKED",
  "count": 100,
  "from": 0
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `status` | string | Фильтр: `"BLOCKED"` |
| `count` | int | Количество элементов |
| `from` | int | Смещение от начала списка |

### Ответ

```json
{
  "contacts": []
}
```

### Особенности

- На момент исследования массив `contacts` пуст (нет заблокированных)
- Возвращает массив контактов в том же формате, что GET_CONTACTS (32)

## GET_USERINFO (opcode 177)

Получение информации о пользователе по ID.

**Важно:** На момент исследования сервер возвращает `cmd=3` с ошибкой
`"Field requirement failed: time"`. Endpoint, вероятно, неактивен или
требует других параметров.

### Запрос

```json
{
  "userId": 6236697,
  "time": 0
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `userId` | int | ID пользователя |
| `time` | int | Всегда `0` (обязательно, без него — ошибка) |
| `fields` | array[string] | Запрашиваемые поля: `"name"`, `"about"`, `"picture"` |

### Ответ

```json
Payload: null (cmd=1)
```

или

```json
{
  "error": "proto.payload",
  "message": "Field requirement failed: time"
}
```

### Особенности

- **Поле `time` обязательно** — без него сервер возвращает `cmd=3`
- Поле `fields` принимается сервером (не вызывает ошибку), но не
  влияет на ответ
- Ошибка `"Field requirement failed: time"` возникает даже при
  наличии всех полей — возможно, endpoint полностью неактивен
- В отличие от других опкодов, возвращает не `cmd=1` с данными,
  а пустой ответ (payload=null) или ошибку

## GET_COMMON_CHATS (opcode 198)

Получение общих чатов с указанным пользователем.

### Запрос

```json
{
  "userIds": [6236697]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `userIds` | array[int] | ID пользователей |

### Ответ

```json
{
  "hasMore": false,
  "commonChats": [
    {
      "owner": 3260455,
      "joinTime": 1,
      "created": 1,
      "lastMessage": { ... },
      "type": "DIALOG",
      "modified": 1781686441819,
      "lastEventTime": 1781686441819,
      "id": 7268926,
      "status": "ACTIVE",
      "participants": { "3260455": ..., "6236697": ... }
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `hasMore` | bool | Есть ещё чаты (пагинация) |
| `commonChats` | array | Массив объектов чатов |

### Особенности

- Формат чатов в `commonChats` идентичен GET_CHATS_BULK (48)
- Позволяет определить, есть ли у вас общий диалог или группа с пользователем

## GET_SESSIONS (opcode 96)

Получение списка активных сессий пользователя.

### Запрос

```json
{}
```

### Ответ

```json
{
  "sessions": [
    {
      "client": "MAX WEB",
      "location": "Russian Federation, Moscow City, IP ...",
      "current": true,
      "time": 1781705573399,
      "info": "Yandex Browser, macOS"
    }
  ]
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `sessions[].client` | string | Название клиента (`"MAX WEB"`, `"MAX Android"`) |
| `sessions[].location` | string | Гео-позиция по IP |
| `sessions[].current` | bool | Текущая сессия |
| `sessions[].time` | int | Timestamp последней активности (мс) |
| `sessions[].info` | string | Информация о браузере/ОС |

### Особенности

- Позволяет увидеть, где ещё выполнен вход в аккаунт
- Поле `current` указывает на сессию, через которую сделан запрос

## GET_BANNERS (opcode 302)

Получение баннеров для главного экрана.

### Запрос

```json
{
  "bannersSync": 0
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `bannersSync` | int | `0` — полная загрузка (дельта-синхронизация, как у FOLDERS) |

### Ответ

```json
{
  "banners": [],
  "updateTime": 1781681068026,
  "showTime": 86400000
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `banners` | array | Массив баннеров (на момент исследования пуст) |
| `updateTime` | int | Timestamp для дельта-синхронизации |
| `showTime` | int | TTL показа баннера в мс (`86400000` = 24 часа) |

### Особенности

- `showTime = 86400000` — баннер показывается не чаще раза в 24 часа
- `updateTime` — при повторном запросе передаётся как `bannersSync` для дельта-обновлений
- Паттерн синхронизации идентичен FOLDERS_GET (272): поле-маркер `{XxxSync: 0}` → серверный timestamp

## CONFIG (opcode 101)

Получение конфигурации сервера (доступные функции, регионы).

### Запрос

```json
{}
```

Поле `interactive` необязательно — подходит любой минимальный payload.

### Ответ

```json
{
  "phone-auth-enabled": true,
  "location": "RU",
  "reg-country-code": ["AZ", "AM", "KZ", "KG", "MD", "TJ", "UZ", "GE"],
  "web-pwa-promo": false
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `phone-auth-enabled` | bool | Доступна регистрация по телефону |
| `location` | string | Код страны пользователя (ISO 3166-1 alpha-2) |
| `reg-country-code` | array[string] | Коды стран, доступных для регистрации |
| `web-pwa-promo` | bool | Показывать промо PWA-версии |

### Особенности

- Работает как после, так и до LOGIN (с `trackId`)
- Возвращает статическую конфигурацию сервера — не меняется в пределах сессии

## GET_BOT_INFO (opcode 106)

Поиск информации о боте по ID.

### Запрос

```json
{
  "botId": 1
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `botId` | int | ID бота (число, не строка) |

### Ответ

Возвращает `cmd=3` с ошибкой `"not.found"`, если бот не найден.
Структура успешного ответа не установлена — на момент исследования
ни один botId (1, 0, 123) не вернул данных.

Формат ошибки:
```json
{
  "error": "not.found",
  "message": "Key: common.finder, args: [tam.bot,БОТ_ID,]."
}
```

### Особенности

- Внутренний поиск через `common.finder` по ключу `tam.bot`
- Требует числовой `botId` (строка вызывает `"Expected number"`)
- Список существующих botId не установлен

## GET_CONTACTS_V2 (opcode 144)

Получение контактов чата. Находится в диапазоне push-опкодов (128–156),
но в отличие от них работает и как запрос (клиент → сервер).

### Запрос

```json
{
  "chatId": 7268926
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `chatId` | int | ID чата |

### Ответ

```json
{
  "contacts": {}
}
```

На момент исследования возвращает пустой объект контактов для всех тестовых
чатов. Возможно, endpoint выполняет поиск контактов только для специальных
типов чатов.

### Особенности

- В опкод-диапазоне push-уведомлений (128–156), но работает как обычный запрос
- В ответе ожидается массив контактов, аналогичный GET_CONTACTS (32)

## PRESENCE_GET (opcode 145)

Получение статуса присутствия бота/пользователя. Находится в диапазоне
push-опкодов (128–156), но работает и как запрос (клиент → сервер).

**Исправление:** Ранее документировался payload `{"interactive": true}`,
однако тесты показывают, что сервер требует поле `botId`.

### Запрос

```json
{
  "botId": 106
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `botId` | int | ID бота для проверки присутствия |

### Ответ

```json
{
  "presence": {
    "seen": 1781932900,
    "on": "ON",
    "status": 1
  },
  "userId": 14032353,
  "time": 1781932900446
}
```

| Поле | Тип | Описание |
|------|-----|----------|
| `presence.seen` | int | Timestamp последней активности (сек) |
| `presence.on` | string | Статус: `"ON"` — онлайн |
| `presence.status` | int | `1` — онлайн |
| `userId` | int | ID пользователя |
| `time` | int | Timestamp уведомления (сек) |

### Особенности

- **Уникальный опкод**: возвращает ответ в формате push (cmd=0), а не cmd=1
- Возвращает присутствие **одного случайного пользователя** — вероятно,
  это механизм опроса/подписки на обновления присутствия
- userId меняется при каждом запросе — сервер выбирает случайного
  активного пользователя
- С полем `chatId` требует `botId`, что указывает на два режима работы:
  общий опрос (interactive) и запрос по боту (botId)
